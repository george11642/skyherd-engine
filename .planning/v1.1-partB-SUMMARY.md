## SSE Event Contract (coded defensively against Part A)

The UI subscribes to two new events that Part A will emit. Until Part A lands, handlers no-op and the legacy `agent.log` fuzzy-match behavior in `ScenarioStrip` continues to detect scenario transitions from log text.

```ts
// scenario.active — fired when a scenario begins (one per pass)
{
  name: string,          // e.g. "coyote", "sick_cow", "water_drop"
  pass_idx: number,      // 0-indexed pass counter within the scenario set
  speed: number,         // playback speed multiplier (1.0 = realtime)
  started_at: string,    // ISO-8601 timestamp, used by breadcrumb elapsed counter
}

// scenario.ended — fired when a scenario completes or fails
{
  name: string,
  outcome: "passed" | "failed",
}
```

Consumers:
- `ScenarioStrip` (B4): applies `.scenario-active` breathing glow to the matching pill.
- `ScenarioBreadcrumb` in `RanchMap` (B4): renders `SCENARIO: {NAME} · mm:ss` chip at top-right of the canvas; ticks elapsed seconds from `started_at` every 1 s; hides on `scenario.ended`.

Both subscribe to `scenario.active` / `scenario.ended` via `getSSE()` — which already flows through `SkyHerdSSE._open()`; the event names must be registered in `web/src/lib/sse.ts` `eventTypes` array when Part A is merged. (Left untouched in Part B to avoid SSE race conditions before Part A ships.)

**Defensive behavior:** If Part A never emits these events, the handlers never fire, `ScenarioStrip` keeps using its legacy `agent.log` heuristic, and `ScenarioBreadcrumb` simply never mounts DOM. Zero-regression fallback.
