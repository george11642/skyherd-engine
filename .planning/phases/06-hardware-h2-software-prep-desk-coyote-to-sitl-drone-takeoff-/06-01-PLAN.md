# Plan 06-01 — pi_to_mission.py bridge (Pi → FenceLineDispatcher → drone)

## Objective

Add `src/skyherd/edge/pi_to_mission.py`: a single-responsibility async subscriber that
takes MQTT events from Phase-5 emitters (`predator.thermal_hit`, `fence.breach`,
`thermal.reading`), dispatches them through `FenceLineDispatcher` (simulation path,
no Anthropic key required), and executes the resulting `launch_drone` tool call
against a real `DroneBackend`.  Every externally-observable hop emits one
`attest.ledger.Ledger.append` entry, preserving the phase-wide attestation chain.

## Context

- `src/skyherd/edge/picam_sensor.py` and `src/skyherd/edge/coyote_harness.py` publish to
  `skyherd/{ranch}/thermal/{cam_id}` and `skyherd/{ranch}/alert/thermal_hit` today but
  **no consumer** wires those events to a drone backend outside simulation.
- `src/skyherd/agents/fenceline_dispatcher.py` `handler()` accepts a `wake_event` and returns a
  tool-call list.  `agents/simulate.fenceline_dispatcher` produces a deterministic
  list without needing a Session's full Claude SDK state — we can build a
  throwaway Session in-process.
- `src/skyherd/mcp/drone_mcp.py` owns the `launch_drone` closure — we can call the
  underlying `DroneBackend.takeoff` + `DroneBackend.patrol` + `DroneBackend.return_to_home`
  directly and match the MCP tool's output shape, or invoke the MCP tool through its
  closure.  Direct backend calls are simpler and keep the pi_to_mission module
  Claude-SDK-free.
- `src/skyherd/attest/ledger.py` has the canonical append API. `Ledger.open(path, signer, ts_provider)`
  gives us full control; `ts_provider` injection keeps determinism.

## Implementation

### File 1 — `src/skyherd/edge/pi_to_mission.py`

Public class `PiToMissionBridge` with:

```python
class PiToMissionBridge:
    def __init__(
        self,
        *,
        ranch_id: str | None = None,
        drone_backend: DroneBackend | None = None,
        ledger: Ledger | None = None,
        agent_session: Session | None = None,
        dispatch: Callable[[dict[str, Any], Session], list[dict[str, Any]]] | None = None,
        mqtt_publish: Callable[[str, bytes], Awaitable[None]] | None = None,
        ts_provider: Callable[[], float] | None = None,
    ) -> None: ...
```

### Behaviour

1. Subscribe-side: `handle_event(topic: str, payload: dict)` — directly callable from tests.
   - **Filter** topics to `skyherd/{ranch_id}/fence/+` or `skyherd/{ranch_id}/alert/thermal_hit` or `skyherd/{ranch_id}/thermal/+`.  Silently drop anything else.
   - **Normalise** the wake event shape expected by `FenceLineDispatcher`:
     - Extract `ranch_id`, `segment` / `cam_id`, `lat`, `lon` with safe fallbacks.
     - For `alert/thermal_hit` payloads, derive `type="thermal.hotspot"` + `segment=entity`.
     - For `fence/...` payloads, derive `type="fence.breach"` + `segment` from the last topic path segment.
   - **Ledger.append**: `source="pi_to_mission", kind="wake_event", payload={topic, normalised_event}`.
2. Dispatch-side: call `self._dispatch(wake_event, self._session)` (default = `agents.simulate.fenceline_dispatcher`).  Get back a list of tool calls.
3. Execute-side: for each tool call, do what the real MCP tool would do, against `self._drone_backend`:
   - `launch_drone` → `await backend.takeoff(alt_m=…)` + `await backend.patrol([Waypoint(...)])`.  Generate a mission_id (UUID4 truncated to 8 hex, seeded via `self._rng` for determinism in tests).  Ledger.append `kind="mission.launched"`.
   - `play_deterrent` → `await backend.play_deterrent(tone_hz=…, duration_s=…)`.  Also publish an MQTT side-emit `skyherd/{ranch_id}/deterrent/play` (picked up by `speaker_bridge` in Plan 06-02).  Ledger.append `kind="deterrent.played"`.
   - `get_thermal_clip` → best-effort `await backend.get_thermal_clip(...)`; ledger kind="thermal.clip_captured".  Non-critical — swallow errors.
   - `page_rancher` / unknown tools → ledger kind="tool.skipped" with the tool name; don't crash.  The actual page_rancher path lives in Phase 3 voice module and is out of scope here.
4. Post-mission: after the loop completes, if the backend is `in_air`, **do NOT** auto-RTL — RTL is a deliberate failover / chaos-monkey path covered in Plan 06-04.  We leave the drone at the waypoint so the hardware demo shows it hovering.

### Constructor defaults (DI pattern)

- `drone_backend=None` → `get_backend()` (env-configured, default `"sitl"` → `"stub"` in tests).
- `ledger=None` → construct with `Signer.generate()` + in-memory `sqlite3.connect(":memory:")` via `Ledger.open(":memory:", signer, ts_provider)`.
- `agent_session=None` → build a throwaway `Session(id=uuid4(), agent_name="FenceLineDispatcher", agent_spec=FENCELINE_DISPATCHER_SPEC)` — no Claude SDK calls, the simulation path doesn't need tokens.
- `dispatch=None` → `agents.simulate.fenceline_dispatcher`.
- `ts_provider=None` → `time.time`, but **only read inside methods**, never at module import.

### Async runtime

- `async def run(self) -> None` — best-effort `aiomqtt` subscriber that connects to `MQTT_URL`, subscribes to the three wildcard topics, and forwards each payload to `handle_event`.  Same pattern as `CoyoteHarness.run`: SIGINT/SIGTERM → `self.stop()`, JSON decode tolerant, swallow publish errors.
- `async def handle_event(topic, payload_dict) -> list[dict]` — the pure, test-friendly core that returns the executed tool-call list.

### File 2 — `tests/edge/test_pi_to_mission.py`

Minimum coverage targets on pi_to_mission.py: ≥85%.  Use `StubBackend` for all tests.

Test classes:

1. `TestTopicFilter` — `handle_event` drops topics outside the three allowed prefixes; `handle_event` returns [] for filtered-out topics; ledger unchanged.
2. `TestFenceBreachFlow` — publish a canonical `fence.breach` payload → expect `launch_drone` + `play_deterrent` + `page_rancher` tool calls; `StubBackend.state` shows `in_air=True`, `mode="AUTO"`; ledger contains entries: `wake_event`, `mission.launched`, `deterrent.played`.
3. `TestThermalHitFlow` — publish a `predator.thermal_hit` payload → expect same cascade as above.
4. `TestDeterrentSideChannel` — assert a `skyherd/{ranch}/deterrent/play` MQTT side-emit was published (via injected `mqtt_publish`).
5. `TestAttestationChain` — after N back-to-back events, `ledger.iter_events()` yields seq=1..N in order; each `prev_hash` chains to previous `event_hash`; `signer.verify` passes on every entry.  Freeze `ts_provider = lambda: 1_714_000_000.0` so ts_iso strings are deterministic.
6. `TestDeterminism` — run the full flow twice with the same seed → payload canonical JSON bytes are byte-identical (after timestamp sanitization).  Use `CoyoteHarness(seed=42, ts_provider=_fixed_ts)` as the event source.
7. `TestBackendFailure` — `StubBackend` with `STUB_FORCE_UNAVAILABLE=True` → `handle_event` catches `DroneUnavailable`, writes a `mission.failed` ledger entry, does **not** raise.  Keeps the subscriber loop alive.
8. `TestNonBreachEventsNoOp` — publish a `water.low` or `sensor.heartbeat` wake → no tool calls, no ledger append beyond `wake_event.ignored`.
9. `TestDetailedWaypoint` — assert the Waypoint sent to `backend.patrol()` has `lat`/`lon`/`alt_m` matching the inbound wake event.

Additional smoke:

10. `TestRunLoopBestEffort` — monkey-patch `aiomqtt.Client` to raise `ConnectionRefusedError`; `asyncio.wait_for(bridge.run(), timeout=0.1)` returns cleanly (or `asyncio.TimeoutError`), no unhandled exceptions.

Use `InMemoryBroker` from `tests/hardware/test_h1_mqtt_bridge.py` — promote it to a shared fixture file if needed, but simpler: re-implement the 40-LOC router inline as a test fixture in this file for zero coupling.
