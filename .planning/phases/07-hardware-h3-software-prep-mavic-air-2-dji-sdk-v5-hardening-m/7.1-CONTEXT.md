# Phase 7.1 — Laptop Drone Control (no-mobile-app path)

**Extension of:** Phase 7 (H3 — Mavic Air 2 hardening)
**Target:** make the laptop the canonical Mavic controller for Friday's field test — zero Mac, zero phone required.
**Deadline:** 2026-04-26 20:00 EST.

---

## Why decimal (7.1 not 11)

George flipped the mobile-app path Thursday afternoon: *"instead of mobile app could we just do form computer?"*  Phase 7 already ships the wiring (`DRONE_BACKEND=mavic_direct` → `MavicBackend`; `DRONE_BACKEND=mavic` → `MavicAdapter` with MAVSDK failover). What's missing is the **operator path** — a laptop-first manual-override panel, documented USB-C cable setup, and preflight updates that promote laptop above iOS/Android.

A decimal phase keeps the change localized to the H3 surface without reshuffling the roadmap three days before submission.

## Scope (three plans)

### 7.1-01 — Backend manual-override API + MAVSDK-over-USB-C smoke

- New module `src/skyherd/server/drone_control.py` — 5 POST endpoints (`/api/drone/arm`, `/takeoff`, `/rtl`, `/land`, `/estop`) wired into `create_app()`.
- Token auth via `SKYHERD_MANUAL_OVERRIDE_TOKEN` env var + `X-Manual-Override-Token` header. Missing token → 401; wrong token → 403; no drone attached → 503.
- Idempotency: double-arm is a no-op; double-estop is safe; takeoff only from armed+grounded.
- Emit SSE event `drone.manual_override` with `{action, actor, ts, success, latency_ms}` after each successful call.
- New tests `tests/server/test_drone_control.py` (token auth, 503 no-backend, idempotency, SSE emit) + `tests/hardware/test_laptop_drone_control.py` (mocked MAVSDK smoke: agent command → backend response → ledger entry).
- Coverage ≥85% on the new drone_control module.

### 7.1-02 — LaptopDroneControl dashboard panel

- New `web/src/components/LaptopDroneControl.tsx` — ARM/DISARM/TAKEOFF/RTL/LAND/EMERGENCY_STOP buttons, hold-to-arm (3 s press) on ARM + TAKEOFF + ESTOP, live state chip band (battery %, GPS valid, signal, mode).
- Gating: only renders when `window.__DRONE_MANUAL_ENABLED === true` OR when a `?drone=1` query param is present on the dashboard URL. Defaults to hidden so judges don't accidentally mash ESTOP.
- Mount as a 5th tab in the right-rail accordion (`App.tsx`), label "Laptop Drone", sage/dust variant to match design.
- Style uses existing tokens: `var(--color-bg-1)`, `var(--color-line)`, `chip chip-sage|chip-danger|chip-muted`.
- `LaptopDroneControl.test.tsx` — ARM button triggers `POST /api/drone/arm` with token header; hold-to-arm requires 3 s; ESTOP works without hold; SSE `drone.manual_override` updates the state display; ARIA labels present.
- New polling hook: fetches `/api/snapshot` every 1 s for drone state (reuses existing endpoint — no new backend work).

### 7.1-03 — Docs + preflight reorder + submission honesty

- New `docs/LAPTOP_DRONE_CONTROL.md` — USB-C cable spec, Mavic RC port (NOT the drone directly), Linux udev rules, WSL2 `usbipd` notes, troubleshooting, `make laptop-drone-smoke` target.
- `docs/PREFLIGHT_CHECKLIST.md` — add Group 6 ("Laptop Drone Control path"), promote laptop to primary, mark iOS/Android as optional.
- `docs/HARDWARE_H3_RUNBOOK.md` — reorder: lead with §1 "Laptop path (primary)", relegate companion-app sections to §9 "Alternative: phone-based control".
- `docs/SUBMISSION.md` — add a 1-paragraph note under "Hardware path" explaining the iOS code is complete but not installed for demo; laptop is canonical.
- New Makefile target `laptop-drone-smoke` → runs `tests/hardware/test_laptop_drone_control.py` with `--no-cov`.

## Hard constraints

- **Determinism:** `make demo SEED=42 SCENARIO=all` MUST remain byte-identical. Manual override endpoints are a no-op (503) when no drone backend is attached, which is the case during scenario playback.
- **Coverage:** project-wide ≥80%, new `drone_control` module ≥85%.
- **Bundle:** ≤90 kB gzip on main index chunk (current ~71 kB; budget ~3 kB for new panel).
- **No new runtime deps** — reuse `httpx` for tests, FastAPI for the endpoints, React + Tailwind tokens for the panel.
- **MIT + zero attribution** — global git config enforces attribution off.
- **No real flight in tests** — all flight paths mocked; real USB-C plug happens Friday.
- **Security:** token-gated endpoints even though bound to localhost. No SSE backpressure regression.

---

## Non-goals

- No real MAVSDK binary install — if missing, tests mock `pymavlink.mavutil.mavlink_connection`. Documented in `docs/LAPTOP_DRONE_CONTROL.md`.
- No iOS/Android deletion — keep code in `ios/` + `android/` but move out of primary runbook.
- No new backend registry entries — `mavic_direct` already exists.
- No dashboard redesign — just a new accordion tab.

## Risks + mitigations

| Risk | Mitigation |
|------|------------|
| WSL2 `usbipd` mid-test flaky | Document as "manual verify Friday"; tests mock everything below the `PymavlinkBackend` socket. |
| MAVSDK missing in CI | Token-gated endpoints return 503 when backend `connect()` fails; smoke test uses `MagicMock` for `mavutil.mavlink_connection`. |
| Judges click ESTOP | Panel hidden by default (`window.__DRONE_MANUAL_ENABLED` or `?drone=1`); ESTOP requires hold-to-fire. |
| Bundle creep | Panel is plain React — no new deps; tokens from CSS. |
| Determinism regression | No code path touched in sim scenarios. Unit test asserts the `/api/snapshot` shape is unchanged. |
| Running out of time (Friday deadline) | 3 plans ≈ 3-4 hours budget; docs can ship minimal + iterate. |
