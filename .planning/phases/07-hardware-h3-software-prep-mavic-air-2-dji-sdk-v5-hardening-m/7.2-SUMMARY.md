# Phase 7.2 Summary — iOS Swift hardening + iOS↔Android parity

## One-liner

Three parallel audits found 28 latent issues in the iOS+Android companion
apps and iOS build system. 7.2 lands all 28 fixes across 5 commits, adds
a new `LostSignalWatchdog.swift` (30 s RTH grace), restores iOS↔Android
parity on battery/wind/geofence/topic-scheme, and makes the CI actually
build the device slice + package an unsigned IPA for Sideloadly — so the
phone-based path is a real secondary option, not aspirational documentation.

## Friday workflow (phone path, now honest)

```bash
# iOS (secondary — needs Mac or IPA from CI)
# Option A: Xcode on a Mac
#   1. Copy Config.xcconfig.template → Config.xcconfig, fill DJI_API_KEY
#   2. Drop DJIWidget.xcframework into ios/SkyHerdCompanion/Frameworks/
#   3. Set SKYHERD_SWIFT_FLAGS=DJI_SDK_AVAILABLE=1 in Config.xcconfig
#   4. xcodegen generate && open SkyHerdCompanion.xcodeproj
# Option B: Sideloadly (no Mac)
#   1. Download unsigned IPA from GH Actions artifact (workflows/ios-app.yml)
#   2. Sideloadly → install on iPhone connected to laptop
#   3. In-app: enter DJI_API_KEY (env var override in Config.swift)

# Android (secondary — stub mode CI green)
#   1. Bench test with SAM Deployment Manager if DJI SDK AAR present
#   2. Otherwise: ./gradlew test passes SafetyGuardsTest + DroneControlEnvelopeTest
```

The laptop path (Phase 7.1) remains canonical. 7.2's job is ensuring the
phone path doesn't lie — if a judge tries to reproduce the iOS build from
README, they should get a buildable IPA.

## Known deferrals

1. **Real DJI SDK xcframework not in repo** — vendor download gate; CI builds
   succeed in stub mode (`SKYHERD_SWIFT_FLAGS` empty → mock `DJIBridge`).
   Runbook documents the drop-in path.
2. **No iOS-on-device CI** — requires a signed build + TestFlight account,
   out of budget for submission. CI runs simulator tests informationally and
   archives an unsigned IPA as artifact.
3. **Android instrumentation tests (real device)** — `SafetyGuardsTest` +
   `DroneControlEnvelopeTest` run in JVM only. Instrumentation tests against
   a real Mavic RC require a paired phone, which is the Friday-field workflow.
4. **Android `DJIWaypointV2Mission`** — same as iOS; both platforms surface
   `E_UNSUPPORTED` and the Python agent treats this as a recoverable nack.
