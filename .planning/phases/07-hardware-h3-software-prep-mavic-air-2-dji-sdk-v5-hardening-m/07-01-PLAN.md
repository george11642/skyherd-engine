# Plan 07-01 — DJI SDK V5 integration audit + inline bug-fix sweep

## Objective

Produce `docs/H3_DJI_AUDIT.md` — a checklist-style audit of both iOS
(`ios/SkyHerdCompanion/`) and Android (`android/SkyHerdCompanion/`)
companion apps covering mission-upload error handling, GPS-denied
behavior, low-battery triggers, lost-signal recovery, and mission-JSON
schema compliance. Fix straightforward Rule-1/Rule-2 bugs inline; log
complex items to `deferred-features.md` (MEDIUM). No file-moving, no
schema changes here — that's 07-03.

## Known gaps (from pre-read)

1. **iOS — `DJIBridge.gotoLocation`** uses `fc.startGoHome` as a stub
   (line 130). This is wrong for a waypoint goto. Not fixed in 07-01;
   log to deferred (requires `DJIWaypointV2Mission` — a day of work).
2. **Android — `DroneControl.cmdGoto`** stubs ACK-ok without calling
   the DJI SDK (line 127–129). Same story — log to deferred.
3. **Android — `DroneControl.getBatteryPercent`** always returns `100f`
   (line 196). **Rule-2 fix:** wire to DJI SDK V5
   `BatteryManager.getInstance().getChargeRemainingInPercent()`;
   fall back to 100f when SDK unavailable. Keeps unit tests green.
4. **Android — `DroneControl.cmdTakeoff`** pulls battery *after*
   constructing the SafetyGuards instance but before checking wind.
   Minor: add a `checkWind` call if the MQTT args contain
   `wind_kt` (optional). **Rule-2 fix.**
5. **Android — lost-signal recovery**: MQTT disconnect currently logs
   but doesn't trigger RTH. **Rule-2 fix:** add a `lostSignalWatchdog`
   that fires `fc.startGoHome` if MQTT stays down for 30 s during
   in-flight state. Gated behind a config flag (`autoRthOnLostSignal`,
   default true).
6. **iOS — `DJIBridge.state`** does not include a `gps_valid` flag
   from `fcState.isGPSSignalStrong` / `gpsLevel`. **Rule-2 fix:** add
   `gps_valid` to `DroneStateSnapshot` and surface it; takeoff should
   refuse when `gps_valid == false` (only in `#if DJI_SDK_AVAILABLE`).
7. **iOS — `DJIBridge.takeoff`** doesn't call `BatteryGuard` — the
   check lives only in `CommandRouter.route`. That's correct layering;
   audit documents it.
