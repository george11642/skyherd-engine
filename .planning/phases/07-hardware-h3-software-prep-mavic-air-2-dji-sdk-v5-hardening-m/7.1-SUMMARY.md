## Friday workflow (no Mac, no phone — the happy path)

```bash
# 1. Power on the Mavic RC controller (NOT the aircraft first)
# 2. Plug USB-C data cable RC → laptop
# 3. Verify enumeration
lsusb | grep -i dji                          # expect a line
ls -l /dev/ttyACM*                           # expect rw-rw----+ group plugdev

# 4. Launch with mavic_direct backend
export SKYHERD_MANUAL_OVERRIDE_TOKEN="$(openssl rand -hex 16)"
DRONE_BACKEND=mavic_direct \
  uv run python -m skyherd.server.live --port 8000 &

# 5. Dashboard + enable manual panel
xdg-open "http://localhost:8000/?drone=1"
# inject token into window.__DRONE_MANUAL_TOKEN via devtools (runbook §5)

# 6. Fly
# → ARM (3 s hold) → TAKEOFF (3 s hold) → RTL (click) → LAND (click)
# → EMERGENCY STOP (3 s hold) if anything looks wrong
```

Runbook cross-links `docs/LAPTOP_DRONE_CONTROL.md` for udev rules, WSL2
`usbipd` steps (not validated in sandbox — manual verify Friday morning),
and MAVSDK wheel troubleshooting.

## Key decisions

| Decision | Rationale |
|----------|-----------|
| Hide panel by default behind `window.__DRONE_MANUAL_ENABLED` OR `?drone=1` | Judges browsing the dashboard during demo must not see an ESTOP button and accidentally mash it. Opt-in gate keeps the UI honest. |
| Hold-to-fire (3 s) on ARM/TAKEOFF/ESTOP only | DISARM/RTL/LAND are *safer* actions — requiring a hold on them creates friction when the rancher genuinely wants to stop. Follow same taxonomy as DJI GO 4's long-press confirmations. |
| ESTOP is always 200 + `best_effort: true` on fallback | Panel never looks broken; the laptop can't make things worse once RTL fails → disconnect at least forces the drone into its own failsafe. |
| Token via env + header (not cookie, not query param) | Localhost-bound still — but a stray `?token=...` in browser history is a known footgun. Header ensures it never hits referrer logs. |
| Keep iOS/Android code + CI (don't delete) | Phase 7.2 hardens them for parity/docs reality; real-drone validation Friday might still exercise the iOS path as a bench-top comparison. Code stays, runbook lead just flips. |
| `mavic_direct` kept as the backend mode | Already existed (Phase 7 H3-02). No new registry entry needed — just a new HTTP surface and a panel on top. |

## Known deferrals

1. **WSL2 `usbipd` path not validated in sandbox** — documented as "manual verify Friday morning" in `docs/LAPTOP_DRONE_CONTROL.md` §5. George runs Ubuntu natively; WSL2 is secondary documentation.
2. **Token injection via devtools `window.__DRONE_MANUAL_TOKEN`** — no login UI; Friday-only path, single-user workstation. Cookie-based auth would be overengineering for a 2-day window.
3. **No real MAVSDK binary install test** — hardware smoke test mocks `pymavlink.mavutil.mavlink_connection` via `MagicMock`. Real plug happens Friday; the MAVSDK wheel ships its own `mavsdk_server` so no apt install required (documented).
4. **Bundle budget 15 kB away from ceiling** — next big dashboard addition may need to compress panels. Not a 7.1 problem.
