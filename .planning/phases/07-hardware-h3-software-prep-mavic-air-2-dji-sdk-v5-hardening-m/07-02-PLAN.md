# Plan 07-02 — MavicAdapter MAVSDK failover backend

## Objective

Ship `src/skyherd/drone/mavic_adapter.py` — a `DroneBackend` subclass
that wraps two inner backends (DJI via `MavicBackend`, MAVSDK via
`PymavlinkBackend`) and fails over at connect-time *or* mid-mission.
Register it as `DRONE_BACKEND=mavic` (replacing the bare MavicBackend)
while preserving `DRONE_BACKEND=mavic_direct` for the raw DJI-only
path. Coverage ≥ 85 % on the new module.

  `MavicBackend` and `"f3_inav"` → `F3InavBackend`. We insert a new
  registration for `"mavic"` → `MavicAdapter` and re-register
  `"mavic_direct"` → `MavicBackend` so the existing mavic unit tests
  keep passing unchanged.
- `src/skyherd/drone/mavic.py` — `MavicBackend` is fully async and has
  injectable `transport`. Good for inner-leg composition; `MavicAdapter`
  just needs to construct it with the same env-driven defaults.
- `src/skyherd/drone/pymavlink_backend.py` — already ABC-compliant with
  similar signatures; its `connect()` tries TCP-SITL first.
  `MavicAdapter` constructs it with injectable `transport` too.
- The existing `docs/HARDWARE_MAVIC_PROTOCOL.md` ACK shape stays
  identical; the adapter is transparent to pi_to_mission.
- `src/skyherd/attest/ledger.py` provides the `Ledger.append()` API;
  the adapter must emit `adapter.failover` entries to preserve the
  attestation chain.

## Implementation

### File 1 — `src/skyherd/drone/mavic_adapter.py` (new, ~350 lines)

Public class `MavicAdapter(DroneBackend)`:

```python
class MavicAdapter(DroneBackend):
    """
    Two-legged DJI + MAVSDK drone backend with mid-mission failover.

    Leg A (primary): MavicBackend — DJI SDK via companion app.
    Leg B (fallback): PymavlinkBackend — MAVSDK over USB-C OTG.

    On connect, tries Leg A first; if unreachable within
    ``connect_timeout_s``, tries Leg B. If both fail, raises
    ``DroneUnavailable``.

    On every actuator call (takeoff, patrol, return_to_home,
    play_deterrent, get_thermal_clip, state), the call is forwarded to
    the active leg. If the active leg raises ``DroneError`` (including
    ``DroneUnavailable``), the adapter records the failover event and
    switches legs once; the call is retried on the other leg. A second
    failure propagates.

    Ledger events emitted:
      - adapter.leg_selected  (at connect: active_leg)
      - adapter.failover      (mid-mission: from_leg, to_leg, reason)
      - adapter.both_legs_failed (terminal)
    """

    def __init__(
        self,
        *,
        primary: DroneBackend | None = None,
        fallback: DroneBackend | None = None,
        ledger: Ledger | None = None,
        ts_provider: Callable[[], float] = time.monotonic,
        connect_timeout_s: float = 3.0,
    ) -> None: ...

    async def connect(self) -> None: ...
    async def takeoff(self, alt_m: float = 30.0) -> None: ...
    async def patrol(self, waypoints: list[Waypoint]) -> None: ...
    async def return_to_home(self) -> None: ...
    async def play_deterrent(self, tone_hz: int = 12000,
                             duration_s: float = 6.0) -> None: ...
    async def get_thermal_clip(self, duration_s: float = 10.0) -> Path: ...
    async def state(self) -> DroneState: ...
    async def disconnect(self) -> None: ...

    # Introspection
    @property
    def active_leg(self) -> str: ...           # "dji" | "mavsdk" | None
    @property
    def failover_count(self) -> int: ...
```

Key behavior:

- `_invoke(method_name, *args, **kwargs)` helper: calls the active
  leg's method, catches `DroneError`/`DroneUnavailable`, triggers
  `_failover(reason=str(exc))`, retries once.
- `_failover`: swaps `_active` ↔ `_other`, increments counter, writes
  `adapter.failover` ledger entry with `prev_leg`, `new_leg`, `reason`,
  `ts=ts_provider()`.
- `_failover` is idempotent *per call*: if the other leg also fails,
  raise `DroneError("Both legs failed: dji={dji_err} mavsdk={mav_err}")`.
- Ledger is optional; when `ledger is None`, failover still happens but
  no entry is written.

### File 3 — `tests/drone/test_mavic_adapter.py` (new, ~400 lines)

Test matrix (target ≥ 85 % coverage on `mavic_adapter.py`):

1. `test_connect_prefers_dji_when_available` — both legs mocked; DJI
   connect ok → `active_leg == "dji"`, ledger has `adapter.leg_selected`.
2. `test_connect_falls_back_to_mavsdk_when_dji_unreachable` — DJI
   raises `DroneUnavailable` → MAVSDK succeeds → `active_leg ==
   "mavsdk"`, ledger records fallback reason.
3. `test_connect_raises_when_both_legs_fail` — both raise
   `DroneUnavailable` → `DroneError` with both messages.
4. `test_takeoff_delegates_to_active_leg` — active=dji, dji.takeoff
   called once, mavsdk.takeoff not called.
5. `test_takeoff_failover_mid_mission` — active=dji, first call raises
   `DroneError("signal_lost")` → failover → mavsdk.takeoff called →
   success. Ledger has `adapter.failover` entry with reason.
6. `test_patrol_failover_preserves_waypoints` — same mission sent to
   fallback leg after DJI fails on patrol.
7. `test_both_legs_fail_in_takeoff_raises` — both legs raise
   `DroneError`; expect terminal error.
8. `test_failover_count_monotonic` — 3 successive failovers increment
   counter; `active_leg` toggles each time.
9. `test_disconnect_closes_both_legs` — even inactive leg's
   `disconnect` is called once.
10. `test_get_thermal_clip_failover_returns_path` — DJI raises →
    MAVSDK synthesises frame and returns path.
11. `test_factory_registers_mavic_as_adapter` — `get_backend("mavic")`
    returns `MavicAdapter`, `get_backend("mavic_direct")` returns
    `MavicBackend`.
12. `test_deterministic_ts_in_ledger_entries` — injected
    `ts_provider=lambda: 1_700_000_000.0` → ledger entries use exact
    ts.
13. `test_state_failover_returns_snapshot` — state sync on DJI raises →
    falls back to MAVSDK state.
14. `test_play_deterrent_failover` — error in DJI → MAVSDK used.
15. `test_connect_timeout_triggers_fallback` — DJI hangs (asyncio
    sleeps > timeout) → MAVSDK used.

All tests use `asyncio.Event` + mock transports; no real network.
