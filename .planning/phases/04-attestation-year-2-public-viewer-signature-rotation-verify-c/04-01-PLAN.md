# Plan 04-01 — Ledger memver_id + Signer Rotation

## Objective

Extend the Ed25519 signer + Merkle ledger to support:

1. **Optional `memver_id` field** on `Ledger.append(...)` that flows into the canonical-JSON payload (so tampering with the memver pairing breaks the hash). Default `None` (backwards compatible).
2. **Key rotation** — a new `Signer.rotate(old_path, archive_dir)` helper that archives the old private key to `runtime/attest_keys/{timestamp}.pem` (0600), generates a new key at `old_path`, and returns the new Signer.
3. **Archived-pubkey verification** — `Ledger.verify()` already stores the pubkey per row, so post-rotation rows verify against the new key naturally; pre-rotation rows verify against the archived pubkey on their own row (no archive lookup needed — the pubkey is already on the row). We add a unit test proving this works end-to-end: start with key A → append 3 events → rotate to key B → append 3 more events → `verify()` returns `valid=True, total=6`.

## Tasks

1. **[auto]** Add `memver_id: str | None = None` parameter to `Ledger.append()`. If provided, inject into `payload` before canonical-JSON hashing as key `_memver_id`. (Prefix with `_` to avoid colliding with user payload keys.) Expose field on the `Event` model as `memver_id: str | None`.
2. **[auto]** Add `Signer.rotate(current_path, archive_dir)` class method — archives existing key by copying to `archive_dir / f"{utc_iso_timestamp}.pem"` mode 0600, generates new Signer, saves to `current_path`, returns new Signer.
3. **[auto]** Update `memory_hook.py` to pass `memver_id=memory.memory_version_id` to `ledger.append()` (parallel to the existing payload field — belt-and-suspenders — so canonical-JSON hashing binds the memver_id).
4. **[auto]** Add tests to `tests/attest/test_signer.py` and `tests/attest/test_ledger.py`:
   - `test_rotate_archives_old_key_and_creates_new`
   - `test_append_with_memver_id_persists_field`
   - `test_rotation_mid_chain_verify_still_valid` (integration: A → 3 events → rotate → B → 3 events → verify).
